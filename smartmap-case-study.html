SmartMap is a deployed UAE government spatial platform used by two very different user groups:

  • Residents finding essential services
  • Government administrators managing spatial data

Both rely on the same interface, creating a challenge of balancing simplicity with complexity.

UX Designer · I Spatial Techno Solutions · Nov 2025–Mar 2026
5 months · Team of 10 · Figma + FigJam · UAE government
3-person design team · I owned research, interaction design, and visual design

The problem

The platform had a layer system that made sense to the engineers who built it and no one else. Nine ungrouped categories, all visible at once, with toggle states that looked identical whether active or not. Residents had no way to browse by intent. The live bed availability data — arguably the most valuable thing on the platform — was buried so deep that none of the 4 users in our research sessions found it without being told it existed.

We were watching people tap around a map hoping something useful would appear. On a platform designed to help them find healthcare.

The harder problem was that government administrators needed the opposite experience — full layer control, specialist data access, precision toggles. One interface. Two completely different mental models. That was the actual design challenge.

What we changed

We tore down the nine flat layers and restructured them into three grouped categories with distinctly visible toggle states. The key insight was that residents think in intent categories like "I need healthcare," not database layer names. We split the primary toolbar to expose only five persistent actions, tucking specialist tools into a contextual secondary panel.

To fix service discovery, we bypassed the layer panel entirely for general users, adding a category filter directly over the map. This allowed the default map state to load entirely clean through progressive disclosure. Services only populate when a user actively clicks a category. Pushing this through required heavy alignment with the PM and the client to ensure performance wasn't compromised.

Map Interface

The default map state is now entirely clean. We utilized progressive disclosure to ensure services only populate when a user actively selects an intent category, reducing cognitive load.

Layer Controls

The restructured layer panel groups nine flat categories into three semantic sets. Users can now instantly distinguish active layers from inactive ones without reading every label.

Search & Discovery

Bypassing the standard layer panel with a dedicated category filter overlay. This makes critical live data, such as hospital bed availability, discoverable in a single click.

In follow-up testing with the same 4 participants, all of them found their target service without assistance — and all 4 found the bed availability data unprompted. In the original interface, none had.

What I took from this

I went in thinking the problem was visual clutter. The research showed the real failure was invisible system state — users had no idea what the map was currently showing them. That's a different problem entirely, and it changes what you design.

We locked the layer architecture before we properly tested with administrator users. Their needs were different enough that we made compromises I'd avoid if I ran it again — with earlier, more deliberate testing across both user types from week one. That's the version I'd build next time.

What I Would Do Next

  • Redesign navigation around user intent
  • Introduce guided discovery
  • Reduce visual noise via progressive layers
  • Improve system feedback

If you're working on something complex that needs to feel simple — I'd like to hear about it.
