AI Data Platform

Designed IST's flagship data intelligence suite — an AI-powered platform that turns raw enterprise data into actionable insights through 8 integrated modules, from data catalogues to spatial analytics.

Lead UX Designer I Spatial Techno Solutions 2025–Present Team of 12 AI / Enterprise SaaS

Context

IST needed a unified product that would serve as the company's core AI offering — a platform where enterprise clients could ingest, catalogue, visualize, analyze, and govern their data from a single interface. The product needed to work across domains: spatial data, sensor networks, operational databases, and document repositories.

I owned the UX architecture and visual design for the entire product — from the marketing landing page to the deep application interfaces of all 8 modules.

The Problem

Eight powerful modules. Zero cohesion. Each was being built by a different engineering pod with its own UI patterns, navigation models, and terminology. Users had to relearn the interface every time they switched modules.

  • Data Catalogues used a tree-based navigation; Map Viewer used a layer panel; Analytics used tabs — same platform, three mental models
  • No shared design system — buttons, cards, and data tables looked different across every module
  • The landing page promised "AI-Powered Insights" but the application felt like 8 separate tools stitched together
  • Admin workflows required navigating between modules to complete a single task

Product landing page: "Transform your Data into AI-Powered Insights" — clear value proposition with module showcase

Design Process

Unified Design Language

Created a shared design system across all 8 modules — consistent navigation patterns, component library, spacing system, and interaction models. Users learn the interface once.

Module Architecture

  • Data Catalogues — Browse and search enterprise datasets with metadata previews, tagging, and access controls
  • Map Viewer — Spatial data visualization with layer management, drawing tools, and basemap switching
  • Data Dictionary — Schema documentation with field-level lineage tracking and relationship maps
  • Admin Module — User management, permissions, audit logs, and system configuration
  • Analytics & Reports — AI-driven insights with custom report builder and scheduled delivery
  • Data Quality — Automated validation rules, anomaly detection, and data health scoring
  • Data Request Portal — Formal data access workflow with approval chains and SLA tracking
  • Doc & Media — Document management with AI-powered search and version control

Left: Data Catalogues with metadata preview. Right: Analytics module with AI-driven insights.

AI Integration — Woven In, Not Bolted On

AI powers three key experiences: smart search across catalogues (natural language queries), automated data quality scoring (anomaly detection on ingestion), and predictive analytics in the reports module. The AI is invisible — users experience better results without having to "use AI" as a separate feature.

Key Decision

The critical decision was choosing pipeline-based navigation over module-based navigation. Instead of 8 sidebar items that felt like separate apps, I organized the interface around the data lifecycle: Ingest → Explore → Analyze → Govern.

Users don't think in modules — they think in tasks. "I need to find a dataset, check its quality, and build a report" is a workflow that crosses three modules. The navigation now reflects that workflow, not the engineering architecture.

Constraint

Each module was at a different development stage. Data Catalogues and Map Viewer were mature; Data Quality and Doc & Media were still in early development. I had to design a system that felt complete even when modules were being added incrementally — the platform needed to feel like a shipped product from day one, not a work-in-progress.

Full platform: unified design language across all 8 modules with consistent navigation and component system

What I Learned

Designing a multi-module product is fundamentally different from designing a single application. The hardest part isn't any individual module — it's the seams between them. When a user switches from Catalogues to Analytics, that transition needs to feel like turning a page, not switching apps.

The best platform UX is invisible infrastructure. Users shouldn't know they're using 8 modules — they should feel like they're using one product that does everything they need.
